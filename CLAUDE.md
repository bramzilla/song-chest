# CLAUDE.md — Song Chest

This file describes the codebase structure, conventions, and development workflows for AI assistants working on this project.

---

## Project Overview

**Song Chest** is a local-first web application for musicians to organize their song ideas, lyrics, and audio recordings. It is a minimal, dependency-light stack: a Python/Flask backend serving a single-page HTML/CSS/JS frontend, with JSON as the only data store.

There is no build step, no bundler, and no external database.

---

## Stack

| Layer    | Technology                  |
|----------|-----------------------------|
| Backend  | Python 3 + Flask >= 3.0.0   |
| Frontend | Vanilla JS + HTML + CSS     |
| Storage  | `data.json` (plain JSON)    |
| Config   | `config.json` (not in git)  |
| Tests    | `test_api.py` (stdlib + requests) |

## Configuration

Song Chest uses `config.json` for runtime configuration. This file is gitignored and must be created by each user:

```bash
cp config-example.json config.json
# Then edit config.json with real paths
```

**`config.json` format:**
```json
{
  "audio_folder": "/absolute/path/to/audio",
  "lyrics_folder": "/absolute/path/to/lyrics"
}
```

If `config.json` is missing, the server auto-creates it using defaults (`./audio` and `./lyrics`).

There are **no environment variables** — all configuration is file-based.

## Data Model

Storage is in `data.json` (plain JSON, no database). The server performs atomic writes using a temp file to prevent corruption.

### Top-level structure
```json
{
  "projects": [],
  "ideas": [],
  "move_history": []
}
```

### Project
```json
{
  "id": "a1b2c3d4",
  "name": "Summer EP",
  "type": "EP",
  "color": "#e8a12c",
  "folder": "summer-ep",
  "created": 1700000000000
}
```

### Idea
```json
{
  "id": "x9y8z7w6",
  "type": "audio",
  "title": "verse sketch",
  "content": "",
  "audiofile": "summer-ep/sketch.m4a",
  "lyricfile": null,
  "project": "a1b2c3d4",
  "tags": ["verse", "draft"],
  "links": ["x1y2z3w4"],
  "notes": "",
  "created": 1700000000000,
  "updated": 1700001000000,
  "auto_imported": true
}
```

**Key conventions:**
- `id`: 8-character UUID prefix, generated by `new_id()` in `server.py`
- `type`: `"audio"` or `"lyric"`
- All timestamps: milliseconds since epoch (JS-compatible)
- `audiofile` / `lyricfile`: relative paths from the configured root folder
- `project`: project `id` or `null` for unassigned ideas
- `links`: bidirectional — maintained automatically by the API on both ends
- `auto_imported`: `true` for files discovered during sync; removed once the user edits the idea

### Move History entry
```json
{
  "id": "h1i2j3k4",
  "ts": 1700002000000,
  "idea_id": "x9y8z7w6",
  "idea_title": "verse sketch",
  "from_project": null,
  "to_project": "a1b2c3d4",
  "moves": [
    {
      "kind": "audio",
      "file": "sketch.m4a",
      "from": "sketch.m4a",
      "to": "summer-ep/sketch.m4a",
      "from_abs": "/path/to/audio/sketch.m4a",
      "to_abs": "/path/to/audio/summer-ep/sketch.m4a",
      "collision": false
    }
  ],
  "undone": false
}
```

## API Reference

All endpoints return JSON. Base URL: `http://localhost:5000`.

### Configuration
| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/config` | Get current config with folder existence flags |
| POST | `/api/config` | Update `audio_folder` and/or `lyrics_folder` |

### Data
| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/data` | Return all projects, ideas, and move_history |

### File Scanning & Sync
| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/scan` | Scan folders and return discovered audio/lyric files |
| POST | `/api/sync` | Auto-import new files as bare ideas |

### Projects
| Method | Path | Description |
|--------|------|-------------|
| POST | `/api/projects` | Create project (also creates disk folder) |
| PUT | `/api/projects/<pid>` | Update project name, type, or color |
| DELETE | `/api/projects/<pid>` | Delete project (orphans its ideas) |

### Ideas
| Method | Path | Description |
|--------|------|-------------|
| POST | `/api/ideas` | Create idea |
| PUT | `/api/ideas/<iid>` | Update idea (handles bidirectional link sync) |
| DELETE | `/api/ideas/<iid>` | Delete idea (removes from all linked ideas) |
| PATCH | `/api/ideas/<iid>/notes` | Update notes field only |

### File Moving
| Method | Path | Description |
|--------|------|-------------|
| POST | `/api/move/preview` | Preview file moves for an idea |
| POST | `/api/move/execute` | Execute file moves with rollback on error |
| POST | `/api/move/undo/<hid>` | Undo a previous move |
| GET | `/api/move/history` | Get last 50 move history entries |

### File Serving
| Method | Path | Description |
|--------|------|-------------|
| GET | `/audio/<path:filename>` | Serve audio file (with optional ffmpeg transcoding) |
| GET | `/lyrics-file/<path:filename>` | Serve a lyrics file |
| GET | `/` | Serve `index.html` |

### HTTP Status Codes
- `200` — Success
- `201` — Resource created
- `400` — Bad request / validation error
- `403` — Path traversal attempt blocked
- `404` — Not found
- `500` — Server error

## Backend Conventions (`server.py`)

- **Atomic saves**: Data is written to a temp file then renamed to prevent corruption on crash.
- **`new_id()`**: Generates 8-character UUID4 prefixes for all IDs.
- **`safe_path(base, rel)`**: Validates that a resolved path stays within `base`; returns `None` on traversal attempts.
- **Bidirectional links**: `PUT /api/ideas/<iid>` reconciles both sides of the `links` array automatically.
- **Audio transcoding**: ALAC/AIFF files are transcoded to AAC via `ffmpeg` if available; cached as `*.transcoded.m4a`.
- **Folder slugs**: Project folder names are kebab-case slugs of the project name (e.g. `"Summer EP"` → `"summer-ep"`).
- **File support**:
  - Audio: `.m4a`, `.mp3`, `.wav`, `.aiff`, `.ogg`, `.flac`
  - Lyrics: `.txt`, `.md`, `.rtf`

## Frontend Conventions (`index.html`)

The entire frontend lives in a single `index.html` file with inline `<style>` and `<script>` blocks. There is no build step.

**Design system:**
- Dark theme with CSS custom properties (`--color-gold`, `--color-crimson`, `--color-teal`, etc.)
- Fonts: `DM Mono` (monospace body), `Fraunces` (serif accents)
- Layout: sticky header, left sidebar for project navigation, main canvas with grid/list toggle

**JS structure:**
- Function-based (no classes, no modules)
- State is kept in a small set of module-level variables (no framework)
- All API calls use `fetch()` with `async/await`
- UI updates are imperative DOM manipulation

**Waveform visualization:**
- Deterministic pseudorandom generation per idea ID — the same idea always renders the same waveform shape.

When editing `index.html`, keep all code in the single file. Do not split into separate files.

## Testing

`test_api.py` is a standalone Python script that hits the live server. It requires `requests` and a running server on port 5000.

Coverage includes: config CRUD, data loading, scan, sync, project CRUD, idea CRUD, linking, notes patching, move preview/execute/undo, history, and edge cases.

**To run:**
```bash
# Terminal 1
python server.py

# Terminal 2
python test_api.py
```

There are no mock-based unit tests; all tests exercise the real HTTP API.

- `config.json` is gitignored — never commit it

### Pull Requests

After every `git push`, always open a pull request. Use `gh pr create` if the GitHub CLI is available. If `gh` is not installed, immediately provide the user with the direct URL to open the PR:

```
https://github.com/bramzilla/song-chest/pull/new/<branch-name>
```

Do not wait for the user to ask — create (or surface) the PR as part of every push.

## What NOT to Do

- Do not add a build step, bundler, or transpiler — the zero-build design is intentional.
- Do not split `index.html` into separate JS/CSS files unless the user explicitly requests it.
- Do not introduce an ORM or database — `data.json` with atomic writes is the intended storage layer.
- Do not add Python dependencies beyond what is in `requirements.txt` without good reason.
- Do not commit `config.json` — it contains user-specific local paths.
- Do not use environment variables for configuration — the app uses `config.json`.
- Do not write non-atomic saves to `data.json` — always use the temp-file-then-rename pattern.

## Common Tasks

### Add a new API endpoint
1. Add a route function in `server.py` using `@app.route()`
2. Follow existing error-handling patterns (return `jsonify({"error": "..."}), 4xx`)
3. Use `save_data()` (not direct `json.dump()`) after any mutation
4. Add test coverage in `test_api.py`

### Add a new UI feature
1. Edit `index.html` — all frontend code lives here
2. Add CSS to the `<style>` block using existing CSS custom properties
3. Add JS to the `<script>` block using plain functions and `fetch()`

### Update the data model
1. Update the relevant section in `server.py` (creation logic)
2. Update `data.json` sample data if appropriate
3. Update this file's **Data Model** section

### Run a full test pass
```bash
python server.py &
python test_api.py
kill %1
```

## Obsidian Integration

An optional read-only integration that scans an Obsidian vault for tagged notes and surfaces them as linkable items in the idea detail view.

### Config shape (`config.json`)

```json
{
  "integrations": {
    "obsidian": {
      "enabled": false,
      "vault_path": "/absolute/path/to/vault",
      "vault_name": "MyVault",
      "filter": {
        "tags": ["song-idea"],
        "folders": []
      },
      "preview_length": 120
    }
  }
}
```

- `vault_path` is stored with `os.path.normpath` (not `Path.resolve()`) to avoid the macOS `/var` → `/private/var` symlink issue.
- If both `filter.tags` and `filter.folders` are empty, all `.md` files in the vault are included.
- The scanner never modifies vault files — read-only throughout.
- Scan depth is hardcoded to 3 levels (`_OBSIDIAN_SCAN_DEPTH = 3`), not stored in config.

### `obsidian_links` data structure (`data.json`)

Stored alongside `projects`, `ideas`, and `move_history`:

```json
{
  "obsidian_links": [
    {"idea_id": "a1b2c3d4", "note_id": "e5f6g7h8"}
  ]
}
```

Each entry links one Song Chest idea to one Obsidian note. The `note_id` is stable across scans (see below).

### Stable ID generation

Each vault note gets an 8-character stable ID derived from its path relative to the vault root:

```python
hashlib.sha1(vault_relative_path.encode()).hexdigest()[:8]
```

This means the same note always receives the same ID regardless of when the vault is scanned.

### API routes

| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/obsidian/notes` | Scan vault and return matching notes |
| POST | `/api/obsidian/link` | Link an idea to a note `{idea_id, note_id}` |
| DELETE | `/api/obsidian/link/<idea_id>/<note_id>` | Remove a link |
| GET | `/api/obsidian/status` | Returns `{enabled, vault_exists, note_count}` |

### macOS symlink note

Always use `os.path.normpath(os.path.expanduser(path))` for `vault_path` — never `Path.resolve()`. macOS exposes `/var` as a symlink to `/private/var`; `resolve()` would rewrite user-provided paths unexpectedly and break path comparisons.
